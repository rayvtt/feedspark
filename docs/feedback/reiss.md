## 2026-07-22 — round 2
### Platform bug (not a content edit) — corrupted live edits after round 1
- Reported live on the deck: a chapter's section header missing, a box rendering with broken margin/position, and unrelated text appearing in a location it doesn't belong (screenshots: a "Review & Project Recap" table row with a giant misplaced text block plus a stray highlighted box, and an "Automation & Pick Readiness" table row that had pulled in an unrelated quote/card from a different chapter).
  → Root cause found and fixed at the platform level, not in the deck content: `assignEids()`/`assignBlockIds()` in `cloudflare/feedspark-deck/src/worker.js` generated every element's persistence id (`data-eid`, block ids, table/group ids) as a single global "Nth element in document order" counter. That's not actually stable across template pushes — round 1's rework added cards/notes to chapters 01/05/06, which shifted the positional index of every element in chapters 06–14, so any edit Ray had already saved against the old index silently re-applied to a *different, unrelated element* on the next load (a stale style→wrong box's margin, stale text→wrong location, a stale delete tombstone→wrong header gone). Confirmed the git-committed deck HTML itself was clean in all three reported spots — this was purely a live-KV/id-scheme issue, not bad markup.
  → Fix: every id counter is now scoped to the nearest preceding chapter (`c1-e3`, `c6-b2`, `c9-g1`, …) via a new `chapterKeyFor()` helper, so editing one chapter can only ever shift ids *within that chapter* — every other chapter's saved edits keep lining up across future pushes. Verified in jsdom: inserting a new card into chapter 6 now produces zero id changes in chapter 14 (previously ~100% would have shifted). A key that no longer matches any element is already a silent no-op on load, so this also self-heals the corruption already reported instead of needing Ray to hit "Reset page" and lose his edits. Shipped in `cloudflare/feedspark-deck/src/worker.js` — applies to every deck, not just Reiss.
  → **Recommend Ray hard-refresh `/deck/reiss` to confirm the three reported spots now render correctly** — if any of them persist after this fix ships, that would point to a second, separate bug rather than the id-scheme issue.
